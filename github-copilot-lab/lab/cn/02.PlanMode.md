## GitHub Copilot Lab

### 什么是 Plan Mode（规划模式）？

GitHub Copilot 的 Plan Mode 是一种在实际编写代码之前，先让 Copilot 探索代码库、收集上下文、明确需求并制定可行实施方案的工作模式。它的核心价值是：

- **先思考再行动**：在动手写代码之前，先理解问题、分析方案
- **减少返工**：通过前期规划，避免实现过程中频繁推翻重来
- **方案透明**：让开发者清楚 Copilot 计划如何实现需求，在实施前进行确认
- **结构化输出**：生成包含具体步骤的实施计划，便于审核和跟踪

### Plan Mode 的适用场景

1. **新增功能**：需要决定代码放在哪里？遵循什么模式？
2. **存在多种方案**：在不同技术、模式或策略之间做选择
3. **修改现有行为**：不确定应该改什么，或可能产生什么副作用
4. **架构决策**：在设计模式或集成方案之间做取舍
5. **跨文件变更**：重构、迁移或横切关注点的修改
6. **需求不明确**：需要先探索才能理解范围

### 在本 Lab 中的应用

在本实验中，我们将使用 GitHub Copilot 的 Plan Mode 来：
- 在旅游网站项目中规划一个新功能的实施方案
- 体验"先规划，再实施"的开发流程

## Lab 步骤

### 第一步：启用 Plan Mode

#### 1.1 目标
了解如何在 GitHub Copilot Chat 中切换到 Plan Mode

#### 1.2 操作步骤

1. **打开旅游网站项目**

   在 VS Code 中打开 Lab 01 创建的旅游网站项目

2. **切换到 Plan Mode**

   在 Copilot Chat 面板中，点击模式选择下拉菜单，将模式从 "Agent" 切换为 "Plan"

   > 💡 **提示**：Plan Mode 下 Copilot 不会直接修改代码，而是会先制定计划并等待你的确认

---

### 第二步：使用 Plan Mode 规划新功能

#### 2.1 目标
使用 Plan Mode 规划如何为旅游网站添加"目的地搜索与筛选"功能

#### 2.2 操作步骤

1. **描述新需求**

   在 Copilot Chat 中（确保已切换为 Plan 模式），输入以下提示词：
   ```
   我想为旅游网站添加目的地搜索和筛选功能。用户可以通过关键词搜索目的地，
   也可以按地区或价格范围进行筛选。请分析当前项目结构并制定实施计划。
   ```

2. **查看规划结果**

   Copilot 会：
   - 自动探索项目中的相关文件
   - 分析当前代码结构
   - 生成一份包含具体步骤的实施计划

3. **审核实施计划**

   仔细阅读 Copilot 生成的计划，关注以下内容：
   - 计划修改哪些文件
   - 新增哪些组件或函数
   - 实施步骤是否合理

4. **提出调整意见**

   如果计划中有不满意的地方，可以继续对话进行调整：
   ```
   我希望搜索功能使用防抖处理，筛选条件改为侧边栏的形式，请更新计划
   ```

5. **确认并执行计划**

   当计划满意后，点击对话中的确认按钮，Copilot 将按照计划开始实施代码变更

#### 2.3 验证
- Copilot 生成了完整的实施计划
- 计划中包含了具体的文件修改列表和步骤说明
- 确认后代码变更符合计划内容

---

### 第三步：对比 Agent Mode 和 Plan Mode

#### 3.1 目标
理解 Plan Mode 与 Agent Mode 的区别和各自适用场景

#### 3.2 操作步骤

1. **切换回 Agent Mode**

   将 Copilot Chat 模式切换回 "Agent"

2. **使用 Agent Mode 执行相同任务**

   在 Agent 模式下输入以下提示词：
   ```
   为旅游网站添加一个"收藏目的地"的功能，用户可以点击收藏按钮将目的地加入收藏列表
   ```

3. **观察执行差异**

   注意 Agent Mode 与 Plan Mode 的区别：

   | 特性 | Plan Mode | Agent Mode |
   |------|-----------|------------|
   | 执行方式 | 先规划，确认后再实施 | 直接开始实施 |
   | 适用场景 | 复杂功能、架构变更 | 明确的小型任务 |
   | 用户控制 | 实施前可审核和调整方案 | 实施过程中可中断 |
   | 上下文收集 | 主动探索代码库 | 边做边收集上下文 |

#### 3.3 验证
- 理解两种模式的执行流程差异
- 能够根据任务复杂度选择合适的模式
